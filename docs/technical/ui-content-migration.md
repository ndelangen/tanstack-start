# UI content migration rollout

The ruleset and faction detail routes are the accepted Mantine application-content pilots. This document records the conventions proved by those pilots and the remaining migration inventory. It is a rollout guide, not authorization to migrate unrelated routes opportunistically.

## Settled pilot conventions

| Concern                    | Convention                                                                                                                                                                                                                                                                                                                |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Theme boundary             | Normal `_app` content uses the shared fixed-light `appContentTheme`. Mantine's layered stylesheet loads before legacy application styles, and the small shell compatibility bridge preserves the existing outer shell. Bare faction-sheet, print, capture, and publisher renderers remain outside the provider and theme. |
| Standard content UI        | Use Mantine components and semantic props directly for layout, typography, surfaces, actions, feedback, overlays, and fields. Start from a Mantine UI composition pattern route-locally; do not recreate the legacy wrapper API under new names.                                                                          |
| Typed navigation           | Keep TanStack Router `Link` as the navigation owner. At Mantine control call sites, use `renderRoot` and spread the supplied root props onto `Link` so styling, accessibility, typed `to`, `params`, and `search` behavior survive.                                                                                       |
| Domain-block exception     | Keep custom markup or CSS only when the block owns Dune Zone behavior, terminology, identity-rich art, or precision-renderer fidelity that a standard Mantine component would erase. Mantine still owns the block's ordinary outer layout and controls; custom styles must not reach into embedded game renderers.        |
| Shared-component migration | Maintain one canonical component. Migrate its standard presentation only when every affected consumer can be checked in the same change; otherwise leave the existing component intact and record it for a later wave. Never introduce parallel Mantine and legacy variants.                                              |
| Shared extraction          | Extract only repeated product semantics with a focused API and representative stories. Repeated use of `Paper`, `Stack`, `Group`, headings, or responsive grids is not sufficient.                                                                                                                                        |

No shared page-level composition was extracted from the pilots. The ruleset route organizes community rules, section navigation, membership, and FAQ workflows; the faction route frames identity-rich leader, troop, and planet lanes plus publishing state. Their similar Mantine surfaces and content hierarchy are system consistency, not a common product concept. Keeping both compositions route-owned avoids a broad page-section wrapper with no stable semantic API. `IconStat` is the narrower exception: it owns only the repeated icon-plus-value-with-tooltip presentation and carries no route semantics.

| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

The shared components have representative stories.                       |

## Remaining application-content waves

Routes are grouped by shared dependencies and migration risk. Within a wave, migrate shared components once and validate all listed consumers. Do not cross the sheet/capture/publishing boundary to retire an application primitive.

| Wave                                      | Routes                                                                                                                                                                                                         | Main legacy reach and reason for grouping                                                                                                                                                                                                                                                                                                                                                                                 |
| ----------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1. Low-risk indexes and static content    | `/`, `/rulesets`, `/factions`, `/profiles`, `/auth`, `/auth/login`, `/auth/error`, `/privacy`, `/assets`, `/assets/create`                                                                                     | Retire straightforward layout/card usage first. The faction index owns `FactionList` (`AutoGrid` and `BlockLink`), which is also embedded by profile detail and must retain its domain visuals. The auth login route reaches `Stack` through `LoginForm`. Auth index, privacy, and asset placeholders already have no frozen imports but still need an intentional Mantine-content pass rather than incidental restyling. |
| 2. Read/detail and operational pages      | `/groups/:groupSlug`, `/profiles/:profileSlug`, `/admin/migrations`                                                                                                                                            | These routes share toolbar, card, button, tooltip, and profile-link patterns. Profile detail also consumes `FactionList`, `ProfileFaqActivity`, and the pilot-migrated `FaqItemList`; validate those shared consumers together.                                                                                                                                                                                           |
| 3. Ruleset, group, profile, and FAQ forms | `/rulesets/create`, `/rulesets/:rulesetSlug/edit`, `/rulesets/:rulesetSlug/faq/create`, `/rulesets/:rulesetSlug/faq/:questionSlug`, `/groups/create`, `/groups/:groupSlug/edit`, `/profiles/:profileSlug/edit` | Migrate settings forms and the common form presentation family together while preserving TanStack Form and validation ownership. FAQ detail combines reading, answers, moderation, and form presentation, so it belongs with the interaction-heavy form wave rather than the simple detail wave.                                                                                                                          |
| 4. Faction editor system                  | `/factions/create`, `/factions/:factionId/edit`                                                                                                                                                                | This is the densest legacy cluster: generic layout/surfaces/buttons plus accordion, popover, tabs, sortable controls, color and text fields. Treat `FactionEditor` and its section components as one bounded migration. Preserve `FactionSheetPreviewIframe` and every embedded game/sheet renderer; the already-Mantine `GroupAssignPopover` remains shared.                                                             |

Intentional non-content exceptions are not migration backlog: `/preview/sheet/:factionSlug` is a bare precision-rendering route, `/auth/oauth` is a non-visual handoff, and the parameter-only ruleset/group route files are outlet owners.

## Per-wave verification

- Exercise every migrated route at narrow, medium, and desktop widths, plus the persistent shell and at least one untouched legacy route.
- Check every consumer of a changed shared component, including routes outside the wave.
- Search migrated routes and new shared content components for frozen `generic/**` and form-presentation imports.
- Search `src/game/**`, faction sheet/print/capture paths, and publisher entry points for Mantine imports; review the changed-path list before running applicable capture and publishing regressions.
- Run typecheck, unit tests, the production application build, Storybook build, scoped formatting, and `git diff --check`.
